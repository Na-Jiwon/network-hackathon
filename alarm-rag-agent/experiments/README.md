# 실험 Ablation 정리

본 문서는 `alarm-rag-agent` 설계 과정에서 수행한 실험들을 **선택의 근거**로 정리한 기록이다.
"왜 이 구성을 최종으로 골랐는가"에 대한 방어 자료.

---

## 1. 임베딩 모델 비교

후보: `all-MiniLM-L6-v2` vs `all-mpnet-base-v2` (둘 다 sentence-transformers 계열).

| 모델 | 차원 | MTEB avg | 모델 크기 | 학습 경보 임베딩 시간* | FAISS 단독 Accuracy** |
|------|-----|----------|-----------|--------------------|----------------------|
| `all-MiniLM-L6-v2`   | 384 | 56.26 | ~80 MB  | 빠름 (기준)        | 0.93 대 |
| `all-mpnet-base-v2`  | 768 | 57.78 | ~420 MB | ~2.5× 느림         | **0.9480** |

\* `load_vectorstore()` 초기화 시간, 로컬 CPU 기준
\** `evaluate_faiss.py` 단건 분류, test 500개 샘플 (`random_state=42`)

**선택: `all-mpnet-base-v2`**

- **이유 1.** 경보 메시지는 `ETH-ERR`, `PSU-FAIL` 같은 짧은 영문 토큰이 많아
  임베딩 품질의 사소한 차이가 최근접 이웃 결정을 뒤집는다. MTEB 1.5pt 차이가 본 도메인에선 체감된다.
- **이유 2.** 추론은 한 번에 1개 경보 분류라 모델 크기로 인한 latency 손해가
  실제 운영 p95(`evaluate_faiss.py` 실측)에서 수십 ms 수준으로 작다.
- **이유 3.** 학습 코퍼스(9,322건)가 작아 초기 임베딩 시간 차이는 수 초 단위로
  무시 가능. 1회성 비용이다.

**버렸을 수 있는 선택:** 한국어 특화 모델(`ko-sroberta-multitask`).
경보 메시지는 거의 전부 영문 약어/코드라 한국어 모델은 오히려 불리했음.

## 2. 프롬프트 언어 (한국어 vs 영어)

같은 의도의 프롬프트를 한/영 두 버전으로 시험했다.

| 구성 | 위치 | 언어 | 사용 지점 |
|------|------|------|-----------|
| 시스템 프롬프트 (ReAct) | `agent.py` `load_agent()` | **영어** | LLM에 역할·도구 호출 규칙 지시 |
| 사용자 프롬프트 (RAG 베이스라인) | `evaluate_faiss.py` | 한국어 | 컨텍스트 + 질문 조립 |
| 호출 시 human 메시지 | `classify_alarm()` / `evaluate_faiss.py` | 한국어 | "경보 메시지 '...'의 장애 유형을 분류해주세요" |

### 관찰

- **시스템 프롬프트는 영어가 안정적.** LLaMA-3.1-8B는 영어 instruction following이
  한국어보다 명확히 강해, "도구를 한 번만 호출하라", "단어 하나로 답하라" 같은
  **제약 조건 준수율**이 영어에서 더 높았다. 한국어로 시스템 프롬프트를 썼을 때는
  도구 반복 호출 / 불필요한 설명 추가 사례가 섞여 나왔다.
- **유저 메시지·컨텍스트는 한국어로 유지.** 경보 원문은 영어지만, 사용 맥락
  ("장애 유형을 분류해주세요")은 한국어가 자연스럽고 정확도 차이도 미세했다.
- **라벨(`LinkCut` / `PowerFail` / `UnitFail`)은 영어 고정.** 한국어 라벨로
  실험했을 때 `extract_label()` 정규화가 복잡해지고, LLM이 "링크 단선" 같은
  변형 표기를 내놓는 실패 모드가 발생했다.

**결론: 시스템 프롬프트 영어, 유저/컨텍스트 한국어, 라벨 영어의 혼합 구성.**

## 3. Threshold 분포 (0.8 / 1.0 / 1.2)

`explore_data.py` 기반 분석. L2 거리 기준, 낮을수록 유사.

### 3-1. 경보 그룹별 거리 분포

| 경보 유형 | 대표 예시 | 가장 가까운 거리 범위 | 해석 |
|-----------|-----------|--------------------|------|
| Train에 존재하는 제조사 A 경보 | `ETH-ERR`, `PSU-FAIL` | 0.55 ~ 0.65 | 정확 매칭 |
| 유사하지만 장애유형 혼재 시작 | —                    | 0.80 ~ 0.90 | 신뢰도 낮음 |
| Train에 없는 B/C사 표현        | `BATTERY_FAIL`, `ETHER_LINK_DOWN` | 1.22 이상 | 미학습 제조사 |

### 3-2. Threshold 후보별 샘플 분포 (k=3 top-1 기준)

| threshold | 필터 통과 비율* | 정확도 (통과분) | 미통과 처리       |
|-----------|----------------|----------------|-------------------|
| **0.8**   | 높음 (제조사 A 대부분) | 매우 높음    | Agent로 fallback  |
| 1.0       | 중간           | 떨어지기 시작  | 혼재 영역         |
| **1.2**   | 거의 전체       | 낮음          | 미학습 제조사까지 긁어옴 |

\* 실제 수치는 `explore_data.py` 실행 결과(거리 스캔)를 근거로 함.

### 3-3. 최종 2단계 구조

```
1차 threshold 0.8  →  신뢰 매칭만 취함 (제조사 A 경보)
      ↓ 결과 없음
2차 threshold 1.2  →  미학습 제조사 경보도 "거친 유사 사례"로 허용
      ↓ 이 시점엔 Agent/LLM이 직접 판단
```

- **0.8만 쓰면:** 미학습 제조사 경보를 전부 놓친다.
- **1.2만 쓰면:** 제조사 A에서도 혼재 영역까지 긁어와 FAISS 단독 정확도가 떨어진다.
- **2단계 fallback:** 제조사 A 는 0.8에서 종결, B/C 는 1.2로 내려가면서
  LLM이 최종 결정. 두 경로 모두 **자기 한계 안에서만 동작**한다.

## 4. 아키텍처 선택 (ReAct Agent vs 단순 RAG)

| 방식 | Accuracy | 장점 | 단점 |
|------|----------|------|------|
| FAISS 단독 (`k=3` 다수결) | **0.9480** | 가장 빠름, 비용 0, 결정적 | 미학습 제조사에서 틀림 |
| 단순 RAG (LLM 1회 호출) | 0.8667 | 설명 생성 가능 | 컨텍스트 비어도 LLM 답변 시도 |
| ReAct Agent (도구 2개)  | 0.8667 | 재검색 전략 동적 | 도구 반복 호출 리스크 |
| **하이브리드 (FAISS 만장일치 + Agent fallback)** | — | FAISS 정확도 + Agent 유연성 | 두 경로 유지 |

**최종: 하이브리드.**

- FAISS k=3 결과가 **만장일치**면 바로 반환 (대부분의 제조사 A 경보).
- 혼재/미매칭이면 Agent가 `refine_search`로 threshold 완화하며 재시도.
- 이 구조는 **저비용 경로 우선 + 고비용 경로는 꼭 필요할 때만**이라는
  운영 관점의 원칙을 그대로 반영한다 (단건 비용은 `../README.md` 운영 지표 참조).

## 5. 재현 방법

```bash
# 임베딩 비교 / threshold 스캔
python explore_data.py

# 4방식 정확도·latency·비용 측정
python evaluate_faiss.py 100    # LLM 샘플 100개

# BM25 + FAISS 하이브리드 실험
python experiments/hybrid_bm25_faiss.py
```

실험 수치는 돌릴 때마다 약간 변한다 (sample `random_state=42` 고정, LLM은 온도·레이트리밋 영향).
위 표의 숫자는 방어 가능한 **범위**로 해석할 것.
